### Indicadores de Domínio por Fase (Gate Criteria)

**Fase 1** — consegue explicar heap/stack/metaspace em entrevista, tunar flags JVM, analisar GC logs reais, escrever código thread-safe demonstrando happens-before.

**Fase 2** — implementa um pipeline de Streams complexo com collectors customizados, resolve exercícios de concorrência com CompletableFuture e Virtual Threads sem data races.

**Fase 3** — cobertura ≥ 85% em projeto real, pipeline de CI com análise estática bloqueante, testes de contrato passando com Pact.

**Fase 4** — consegue explicar o ciclo de vida de um bean Spring, implementar AutoConfiguration customizada, oauth2 Resource Server com roles e método security.

**Fase 5** — build nativo Quarkus funcional com tempo de startup < 50ms, extensão customizada com Dev Service funcionando.

**Fase 6** — implementa Saga Pattern com Kafka + Outbox, explica exatamente-once semantics e compensating transactions.

**Fase 7** — otimiza query com N+1 problem usando batch fetch, explica MVCC do PostgreSQL, implementa cache L2 com Redis e invalidação correta.

**Fase 8** — provisiona infraestrutura com Terraform, configura HPA no K8s, CI/CD completo com rollback automático, traces distribuídos no Grafana.

**Fase 9** — realiza system design de plataforma de pagamentos em 45 minutos com justificativas arquiteturais, faz code review apontando race conditions e vazamentos de memória.

### 🔷 FASE 1 — FUNDAMENTOS SÓLIDOS E LOW-LEVEL JAVA (Semanas 1–6)

---

#### SEMANA 1 — Revisão Cirúrgica de Fundamentos + OOP Profunda

**Tema Central:** Solidificar sintaxe, tipos, controle de fluxo e OOP com qualidade de produção.

**Distribuição das 8h diárias:**
- 2h leitura ativa (Effective Java, capítulos 1–4)
- 3h codificação prática orientada a exercícios
- 1h revisão com Anki (criar cards sobre regras do Effective Java)
- 1h refatoração de código próprio com feedback do compilador e checkstyle
- 1h exercícios no Exercism.org (trilha Java)

**Projeto Prático Obrigatório:**
Implemente um sistema de `Biblioteca Digital` usando apenas Java puro (sem frameworks). Requisitos:
- Herança vs composição: `Publicacao → Livro, Artigo, Revista`
- Interfaces com default methods (`Emprestavel`, `Pesquisavel`)
- Enum para status com comportamento polimórfico
- Custom Exceptions com hierarquia (`BibliotecaException → ItemIndisponivelException`)
- Generics: `Repositorio<T extends Publicacao>`
- Nenhum campo público; encapsulamento total

**Técnica Ultralearning:** *Prática Direta* — toda teoria codificada imediatamente.

**Ferramenta de Feedback:** Checkstyle + compilador + testes JUnit 5 básicos + peer review via ChatGPT/Claude para verificar design.

**Armadilhas Comuns:**
- Usar herança onde composição seria melhor (regra: "É um" vs "Tem um")
- Ignorar a diferença entre `equals/hashCode` e referência de objeto
- Criar hierarquias de exceção sem considerar checked vs unchecked

#### SEMANA 2 — Generics Avançados + Reflection + Annotations

**Tema Central:** Dominar o sistema de tipos avançado do Java e metaprogramação.

**Distribuição das 8h diárias:**

- 1h Anki + revisão spaced repetition das regras de Generics
- 1h Exercism (exercícios de generics) + LeetCode (estruturas com generics)

**Projeto Prático Obrigatório:**
Implemente um **mini framework de injeção de dependências** com:
- Annotation `@Inject`, `@Component`, `@Singleton`
- `ApplicationContext` que varre classpath via Reflection
- Resolve dependências circulares (e lança exceção clara se insolúvel)
- Suporte a `@PostConstruct`
- Testes: valide que beans são criados corretamente

**Técnica Ultralearning:** *Intuição* — ler o código-fonte de frameworks para entender o "porquê" antes de usar.

**Ferramenta de Feedback:** Testes unitários cobrindo todos os casos de uso; comparar implementação com Guice (Google) via leitura de código.

**Armadilhas Comuns:**
- Confundir `? extends T` (produtor) com `? super T` (consumidor) — regra PECS
- Acreditar que Generics existem em runtime (type erasure elimina em bytecode)
- Usar Reflection sem tratamento de `IllegalAccessException` e `ClassNotFoundException`

#### SEMANA 3 — JVM Internals I: ClassLoader, Bytecode, JIT

**Tema Central:** Entender o que acontece com seu código Java antes e durante a execução.

**Distribuição das 8h diárias:**
- 2h teoria: ClassLoader hierarchy, bytecode (javap), JIT phases (C1/C2)
- 2h laboratório: descompilar classes com `javap -c -verbose`, analisar bytecode
- 2h projeto prático (ver abaixo)
- 1h leitura: JEPs relacionados (JEP 295 — AOT, JEP 483 — AOT Java 24)
- 1h revisão e Anki

**Projeto Prático Obrigatório:**
- Crie um **ClassLoader customizado** que carrega classes criptografadas em disco e as decripta em runtime
- Use `javap` para comparar bytecode de: loop for, stream, lambda, método recursivo vs iterativo
- Escreva um relatório (README.md) explicando cada instrução bytecode dos exemplos

**Técnica Ultralearning:** *Drill* — foco cirúrgico em um componente da JVM por vez.

**Ferramenta de Feedback:** Comparar análise de bytecode com material do livro "Inside the Java Virtual Machine" (Venners) + validação com JVM TI agent básico.

**Armadilhas Comuns:**
- Acreditar que Java é lento sem entender que o JIT pode fazer inlining e escape analysis
- Ignorar o double-check locking e seus problemas pré-Java 5

#### SEMANA 4 — JVM Internals II: Garbage Collectors + Tuning

**Tema Central:** Dominar o ciclo de vida de objetos, GC algorithms e tuning de produção.

**Distribuição das 8h diárias:**
- 2h teoria: G1GC regions, ZGC concurrent phases, Shenandoah barriers
- 2h laboratório GC: geração de GC logs reais, análise com GCViewer e GCEasy
- 2h projeto prático (ver abaixo)
- 1h simulação: criar memory leak deliberado e diagnosticar com heap dump (VisualVM/MAT)
- 1h Anki: flags JVM críticas (`-Xms`, `-Xmx`, `-XX:+UseG1GC`, `-XX:MaxGCPauseMillis`, etc.)

**Projeto Prático Obrigatório:**
Crie um **benchmark de GC** com JMH (Java Microbenchmark Harness):
- Aloque objetos de diferentes tamanhos e lifetimes
- Compare G1GC vs ZGC em latência de pause e throughput
- Gere relatório com gráficos de GC pause time usando dados reais dos logs
- Documente as flags usadas e o impacto de cada uma

**Técnica Ultralearning:** *Experimentação* — comparar abordagens com benchmarks reais, não opiniões.

**Ferramenta de Feedback:** JMH results + GCEasy analysis + validação cruzada com documentação Oracle.

**Armadilhas Comuns:**
- Ajustar heap sem considerar o padrão de alocação da aplicação
- Usar `-Xmx` igual a `-Xms` sem entender o custo de heap comprometido
- Ignorar metaspace (OutOfMemoryError: Metaspace é diferente de heap)

#### SEMANA 5 — Java Memory Model (JMM) + Concorrência Profunda

**Tema Central:** Entender a raiz teórica da concorrência Java para nunca escrever código com race conditions.

**Distribuição das 8h diárias:**
- 2h teoria: happens-before, visibility, atomicidade, volatile, synchronized semântica
- 1h leitura: capítulos 1–5 "Java Concurrency in Practice" (Goetz)
- 3h implementação prática de primitivos de concorrência
- 1h análise de bugs de concorrência clássicos (double-check locking pré-Java 5, lazy init)
- 1h Anki + revisão

**Projeto Prático Obrigatório:**
Implemente **do zero** (sem usar `java.util.concurrent`):
- `ReentrantLock` simplificado usando `synchronized` + `wait/notify`
- `BoundedBlockingQueue<T>` thread-safe com capacidade configurável
- `ThreadSafeCounter` com e sem `volatile` — benchmark comparativo
- Demonstre visualmente (com logs) a diferença entre código com e sem `volatile`

**Técnica Ultralearning:** *Prática Direta* — implementar os primitivos antes de usar os prontos.

**Ferramenta de Feedback:** Testes de stress com múltiplas threads (10.000 iterações concorrentes) + análise com Java Flight Recorder.

**Armadilhas Comuns:**
- Confundir atomicidade com visibilidade (são garantias diferentes)
- Usar `volatile` para operações compostas (check-then-act não é atômico)
- Esquecer que `HashMap` em ambiente concorrente pode causar loop infinito em resize

#### SEMANA 6 — Concorrência Avançada: java.util.concurrent + JNI

**Tema Central:** Dominar o toolkit de concorrência de alto nível e entender as fronteiras do Java.

**Distribuição das 8h diárias:**
- 2h teoria: Executors, ThreadPoolExecutor internals, Fork/Join, semáforos
- 2h laboratório: Fork/Join para processamento paralelo de dados
- 2h projeto prático (ver abaixo)
- 1h JNI: conceitos, casos de uso, riscos, alternativas modernas (Panama API)
- 1h revisão de toda a Fase 1 + atualização do Anki deck

**Projeto Prático Obrigatório:**
Implemente um **sistema de processamento de arquivos CSV paralelo**:
- `ThreadPoolExecutor` customizado com política de rejeição específica
- `Fork/Join` para dividir arquivos grandes em chunks e processar em paralelo
- `CountDownLatch` para sincronização de fase de merge
- `CompletableFuture` para orquestração do pipeline async
- Benchmark: compare single-thread vs parallel performance

**Técnica Ultralearning:** *Retenção* — escrever um blog post/README detalhado explicando cada decisão de design.

**Ferramenta de Feedback:** JFR recording da execução, análise de thread contention, comparação com biblioteca rival (RxJava).

**Armadilhas Comuns:**
- Criar `ThreadPool` muito grande (context switch overhead supera ganho de paralelismo)
- Esquecer de tratar exceções dentro de `Callable` (são engolidas silenciosamente em `Future.get()`)
- Usar `ForkJoinPool.commonPool()` para tarefas bloqueantes (paralisa o pool do sistema)

#### SEMANA 7 — Collections Framework Internals

**Tema Central:** Entender o "por dentro" das estruturas de dados para fazer escolhas corretas em produção.

**Distribuição das 8h diárias:**
- 2h teoria: HashMap internals (hash function, bucket, treeification em Java 8+), ArrayList vs LinkedList performance real
- 2h leitura de código-fonte: `HashMap.java`, `ArrayList.java`, `TreeMap.java` no OpenJDK
- 3h implementação prática
- 1h benchmarks com JMH comparando estruturas

**Projeto Prático Obrigatório:**
Implemente um **HashMap do zero** com:
- Hashing, chaining, load factor e resize automático
- Comparar performance com `java.util.HashMap` via JMH
- Demonstrar quando `TreeMap` é preferível ao `HashMap` (com benchmark)
- Implementar `LRUCache` usando `LinkedHashMap` com `removeEldestEntry`

**Armadilhas Comuns:**
- Usar `LinkedList` acreditando que inserção é sempre mais rápida que `ArrayList`
- Iterar sobre `HashMap` e modificá-lo simultaneamente (`ConcurrentModificationException`)
- Usar coleções mutáveis em chaves de `HashMap` (contrato `hashCode` violado)

#### SEMANA 8 — Streams API, Lambdas e Functional Programming

**Tema Central:** Dominar o paradigma funcional em Java com profundidade real.

**Distribuição das 8h diárias:**
- 1h teoria: lazy evaluation, intermediate vs terminal operations, Spliterator
- 2h laboratório: collectors customizados, `Collector.of()`, downstream collectors
- 3h projeto prático
- 1h análise de performance: Stream vs for-loop vs parallel Stream (JMH benchmarks)
- 1h Exercism: exercícios de programação funcional

**Projeto Prático Obrigatório:**
Construa um **pipeline de análise de dados de vendas**:
- Lê dados de CSV (sem biblioteca externa), processa com Streams
- Implementa `Collector<T,A,R>` customizado para estatísticas agregadas
- Usa `groupingBy`, `partitioningBy`, `teeing` (Java 12+)
- Compara performance serial vs parallel Stream em datasets de tamanhos diferentes
- Demonstra onde `parallel()` prejudica ao invés de ajudar

**Armadilhas Comuns:**
- Usar `parallel()` em todo lugar sem medir (overhead > ganho em datasets pequenos)
- Stream com estado compartilhado em operações paralelas (race condition silenciosa)
- `Optional` como campo de classe (não é serializável e não foi projetado para isso)

#### SEMANA 9 — CompletableFuture e Async Programming

**Tema Central:** Programação assíncrona sem bloquear threads, padrão de produção em sistemas modernos.

**Distribuição das 8h diárias:**
- 2h teoria: CompletableFuture API completa, thread pools padrão, exception handling
- 3h projeto prático com chamadas async encadeadas
- 1h comparação com RxJava e Project Reactor (preparação para WebFlux)
- 1h leitura: JEP 425 (Virtual Threads)
- 1h revisão + Anki

**Projeto Prático Obrigatório:**
Construa um **aggregator de dados de múltiplas "APIs" simuladas** (métodos que dormem):
- Busca dados de 5 "fontes" em paralelo com `CompletableFuture.allOf()`
- Timeout por chamada com `orTimeout()`
- Fallback com `exceptionally()` e `handle()`
- Composição de pipelines com `thenCompose()` vs `thenApply()`
- Demonstre diferença entre `thenApply()` (síncrono) e `thenApplyAsync()` (novo thread)

**Armadilhas Comuns:**
- Esquecer de especificar Executor customizado (usa `commonForkJoinPool` por padrão)
- Encadear `thenApply` sem tratar exceções intermediárias (falhas silenciosas)
- Confundir `thenCompose` (flatMap) com `thenApply` (map)

#### SEMANA 10 — Virtual Threads (Project Loom) + Java 16–21 Features

**Tema Central:** Aproveitar as inovações do Java moderno que mudaram a forma de escrever código concorrente e expressivo.

**Distribuição das 8h diárias:**
- 2h Virtual Threads: comparação com platform threads, structured concurrency, `StructuredTaskScope`
- 2h features modernas: Records, Sealed Classes, Pattern Matching (switch expressions, guarded patterns)
- 2h projeto prático
- 1h leitura de JEPs: 444 (Virtual Threads), 441 (Pattern Matching for switch), 409 (Sealed Classes)
- 1h benchmark: Virtual Threads vs CompletableFuture vs Platform Threads em I/O-bound workload

**Projeto Prático Obrigatório:**
Refatore o projeto da Semana 9 para usar Virtual Threads:
- `Thread.ofVirtual().start(...)` e `Executors.newVirtualThreadPerTaskExecutor()`
- `StructuredTaskScope.ShutdownOnFailure` para gerenciamento de escopo
- Modele domínio com Records e Sealed Classes (ex: `Result<T>` como sealed: `Success | Failure`)
- Pattern Matching em switch para processar tipos de eventos

**Armadilhas Comuns:**
- Usar `synchronized` com Virtual Threads (causa pinning — prefira `ReentrantLock`)
- Achar que Virtual Threads substituem reactive programming em todos os casos
- Usar `ThreadLocal` com Virtual Threads sem entender o comportamento de herança

**Projeto Prático Obrigatório:**

- Probes configuradas corretamente (readiness falha durante migração Flyway)
- `PodDisruptionBudget` para zero-downtime deployments

**Armadilhas Comuns:**
- Liveness probe muito agressiva mata pod durante GC pause
- Secrets em `ENV` variáveis visíveis em `kubectl describe pod`
- Sem `resources.limits` = um pod consome todo o nó

## PARTE 4 — ESTRATÉGIAS DOS 9 PILARES DO ULTRALEARNING

### 1. Meta-aprendizagem 🗺️
**Como aplicar:** Antes de cada fase, dedique 2h para mapear:
- O que você precisa aprender nesta fase
- Quais recursos existem (livro, doc, código-fonte)
- Quais são as conexões com fases anteriores
- Qual é o projeto que provará que você aprendeu

**Prático:** Criar um arquivo `LEARNING_MAP.md` no início de cada fase com este mapeamento. Revisar ao fim da fase.

---

### 2. Foco 🎯
**Como aplicar:** Deep Work estruturado em blocos de 90 minutos (não Pomodoro clássico — trabalho técnico profundo requer mais de 25 minutos para entrar em flow).

**Protocolo diário:**
- 8h divididas em: 2 blocos de 90min (teoria/código) + 1h almoço + 2 blocos de 90min (projeto) + 1h revisão
- Ambiente: notificações desligadas, fone com ruído branco, IDE em tela cheia
- Regra: qualquer dúvida no bloco → anota, não interrompe o flow

---

### 3. Prática Direta ⚡
**Como aplicar:** Regra absoluta — **zero teoria sem código em paralelo**.

**Protocolo:**
- Leu um conceito? Implemente em 15 minutos ou menos
- Nunca avance capítulo sem o código do capítulo anterior compilando e testado
- Cada conceito vira pelo menos um teste de unidade que o valida

---

### 4. Drill (Repetição Focada) 🔁
**Como aplicar:** Identificar os pontos mais fracos de cada fase e atacá-los isoladamente.

**Prático:**
- Anki deck por fase com cards técnicos (não conceituais — cards de "complete o código")
- Exercism.org: 5 exercícios por semana na trilha Java
- LeetCode: 3x por semana para manter pensamento algorítmico

---

### 5. Recall 🧠
**Como aplicar:** Testar o próprio conhecimento sem olhar material.

**Prático:**
- Ao fim de cada semana: fechar todos os materiais e escrever o que aprendeu (free writing, 30 min)
- Code-from-memory: reimplementar um padrão/algoritmo sem consultar código anterior
- Flashcards de APIs Java críticas (anotações Spring, métodos de Streams, flags JVM)

---

### 6. Feedback 📊
**Como aplicar:** Usar múltiplas fontes de feedback imediato.

**Fontes:**
- **Compilador:** não ignore warnings — trate como erros
- **Testes:** se falham, o entendimento estava errado
- **SonarQube/SpotBugs:** feedback de qualidade de código
- **Code review com IA:** usar Claude/GPT para revisar código com prompt específico: "Você é um sênior Java engineer. Revise este código identificando: performance issues, design problems, security vulnerabilities, missing test cases"
- **Comunidade:** postar dúvidas no Stack Overflow, r/javahelp, Discord Java Brasil

---

### 7. Retenção 📖
**Como aplicar:** A informação fica apenas quando é ensinada.

**Protocolo:**
- Ao fim de cada fase: escrever um blog post técnico (pode ser privado inicialmente)
- README.md de cada projeto com explicações de todas as decisões arquiteturais
- Tech talks pessoais: gravar vídeo de 10 min explicando o que aprendeu na semana

---

### 8. Intuição 💡
**Como aplicar:** Ler código-fonte dos frameworks para entender "o porquê" antes do "como".

**Protocolo:**
- Antes de usar qualquer anotação Spring, ler o código dela no GitHub
- Quando algo "não funciona como esperado", ler o código-fonte antes de buscar solução no Stack Overflow
- Issues do GitHub do Spring Boot / Quarkus são leitura obrigatória — mostram problemas reais de produção

---

### 9. Experimentação 🧪
**Como aplicar:** Comparar abordagens, medir, decidir baseado em dados.

**Protocolo:**
- Toda afirmação de performance deve ter benchmark JMH que a prove
- Antes de adotar uma solução, implementar 2 abordagens e comparar
- "Qual é o custo real de usar `synchronized` vs `ReentrantLock`?" — meça, não suponha

## APÊNDICE — CHECKLIST ANTIPATTERNS POR FASE

### Fase 1–2 (Java Core)
- [ ] Não usar `==` para comparar Strings
- [ ] Não ignorar exceções capturadas (catch vazio ou só com `e.printStackTrace()`)
- [ ] Não fazer cast sem `instanceof`
- [ ] Não criar ThreadLocal sem remover em `finally`
- [ ] Não usar `Date`/`Calendar` — usar `java.time`

### Fase 3 (Testes)
- [ ] Não nomear testes como `test1()`, `testMethod()`
- [ ] Não usar `Thread.sleep()` em testes — usar `Awaitility`
- [ ] Não compartilhar estado entre testes
- [ ] Não mockar o que você está testando

### Fase 4 (Spring)
- [ ] Não usar `@Autowired` em campo
- [ ] Não colocar lógica de negócio em `@Controller`
- [ ] Não expor entidades JPA diretamente na API
- [ ] Não fazer rollback manual quando `@Transactional` faz automaticamente
- [ ] Não usar `Optional.get()` sem `isPresent()` ou `orElseThrow()`

### Fase 6 (Mensageria)
- [ ] Não consumir mensagem sem idempotência
- [ ] Não fazer commit de offset antes de processar
- [ ] Não ter Consumer sem DLQ configurada
- [ ] Não publicar no Kafka dentro de transação de banco sem Outbox Pattern

### Fase 7 (Persistência)
- [ ] Não carregar todo o dataset em memória para filtrar em Java
- [ ] Não usar `@OneToMany` sem `fetch = LAZY` explícito
- [ ] Não fazer `session.flush()` desnecessário
- [ ] Não usar conexão sem pool em produção

### Fase 8 (Cloud/DevOps)
- [ ] Não hardcodar credenciais em código ou Docker image
- [ ] Não usar `latest` como tag de imagem Docker
- [ ] Não expor Actuator sem autenticação
- [ ] Não fazer deploy sem health check configurado
